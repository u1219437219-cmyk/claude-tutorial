# Claude 前期需求分析与方案设计工作流

## 文档目的

前面那套文档主要解决的是“项目开发阶段怎么和 Claude 协作”。

这份文档补的是更靠前的阶段：

- 需求分析
- 方案设计
- 任务规划

也就是在真正开始写代码之前，怎么用 Claude 帮你把事情想清楚、拆清楚、排清楚。

---

## 一、先记住一条总原则

**前期阶段不要急着让 Claude 写代码，先让它帮你把问题定义清楚。**

最稳的顺序是：

1. 需求澄清
2. 边界收敛
3. 方案比较
4. 设计确认
5. 任务拆分
6. 再进入实现

一句话：

**先想明白，再做出来。**

---

## 二、前期工作流总览

你可以把前期阶段理解成 4 步。

### 第 1 步：需求分析

解决：

- 这个需求到底要解决什么问题
- 目标用户是谁
- 第一版到底要做到什么程度

### 第 2 步：方案设计

解决：

- 技术上怎么做
- 有几种方案
- 各自 tradeoff 是什么

### 第 3 步：任务规划

解决：

- 要分成哪些步骤
- 哪些先做，哪些后做
- 哪些能独立验证

### 第 4 步：交付准备

解决：

- 进入开发前，需要先准备哪些规则和协作方式
- `CLAUDE.md` 要写什么
- skill / agent 要不要先建

---

## 三、需求分析阶段怎么用 Claude

这个阶段最重要的事情不是“想技术”，而是把需求说清楚。

### 目标

让 Claude 帮你输出：

- 用户是谁
- 问题是什么
- 核心目标是什么
- 第一版边界是什么
- 什么先做，什么后做

### 推荐对话模板

```text
我现在有一个需求想法，但还不够清楚。
先不要写代码。

请帮我做需求澄清，输出：
1. 这个需求要解决的核心问题
2. 目标用户是谁
3. 用户场景是什么
4. 第一版必须有的内容
5. 可以后续再做的内容
6. 这个需求里最容易模糊或失控的地方
```

### 适合 Claude 做的事情

- 帮你提炼问题定义
- 帮你识别需求模糊点
- 帮你拆 MVP
- 帮你列出输入条件和成功标准

### 不要一上来让 Claude 做的事情

- 直接选框架
- 直接写代码
- 直接生成一大堆目录

因为这时候边界通常还没清楚。

---

## 四、方案设计阶段怎么用 Claude

当需求已经清楚一些后，Claude 最适合帮你做的是“方案对比”。

### 目标

让 Claude 帮你输出：

- 2 到 3 种可行方案
- 每种方案的优缺点
- 推荐方案及原因

### 推荐对话模板

```text
现在需求边界基本清楚了。
先不要写代码。

请针对这个需求给我 2-3 种实现方案，并从下面几个维度比较：
1. 开发速度
2. 复杂度
3. 可维护性
4. 风险
5. 扩展性

最后给我一个你最推荐的方案，并说明为什么。
```

### 这一阶段最适合问 Claude 的问题

- 这个需求有哪几种实现路径？
- 哪种方案最适合 MVP？
- 哪种方案后期维护成本最低？
- 哪种方案回归风险最小？

### 最重要的提醒

不要只问：

```text
这个需求怎么做？
```

更好的问法是：

```text
请给我多个方案，并明确 tradeoff。
```

这样 Claude 会更像架构搭档，而不是直接拍脑袋给一个答案。

---

## 五、任务规划阶段怎么用 Claude

这是最容易被忽略，但最值的一步。

### 目标

让 Claude 帮你输出：

- 有顺序的小任务
- 每个任务的边界
- 每个任务怎么验证
- 哪些任务可以并行，哪些不行

### 推荐对话模板

```text
现在方案已经基本确定。
先不要直接写代码。

请把这个需求拆成 8-12 个尽量独立的小任务。
要求：
1. 每个任务尽量边界清晰
2. 有合理顺序
3. 标出依赖关系
4. 每个任务都能独立验证
5. 尽量适合小步实现和 review
```

### 如果需求较复杂

可以继续补一句：

```text
请把这些任务分成：
- 基础搭建
- 核心功能
- 收尾验证
```

### 这一阶段的关键价值

不是让 Claude 列“TODO 列表”，而是让它帮你把开发推进节奏设计好。

---

## 六、空白项目怎么用 Claude 做前期工作

如果是从 0 到 1 的项目，推荐顺序更重要。

### 推荐顺序

1. 定义项目目标
2. 拆 MVP
3. 比较技术方案
4. 设计目录结构
5. 生成初始版 `CLAUDE.md`
6. 拆第一批任务
7. 再开始实现

### 一整段推荐开场话术

```text
我要从 0 开始做一个新项目。
先不要写代码，先帮我完成这几件事：
1. 把我的想法整理成清晰的项目目标
2. 拆出 MVP 范围
3. 推荐合适的技术方案
4. 设计最小可扩展的目录结构
5. 生成项目初始版 CLAUDE.md
6. 把第一阶段开发任务拆出来

整个过程请遵循：
- 优先可落地，不要过度设计
- 优先 MVP，不提前做复杂扩展
- 每一步先解释思路，再继续下一步
```

## 七、老项目新增需求怎么用 Claude 做前期工作

如果是老项目，新需求的前期工作重点不是“从零设计”，而是“先理解现有系统对这个需求的约束”。

### 推荐对话模板

```text
这是一个老项目上的新需求。
先不要改代码。

请先帮我分析：
1. 这个需求会影响哪些已有模块
2. 哪些地方可能最容易产生回归
3. 现有结构是否适合直接承载这个需求
4. 是扩展现有逻辑更合适，还是单独加一层更合适
5. 推荐的实现路径是什么
```

### 这类场景最重要的点

Claude 要先理解“已有约束”，再谈方案。

---

## 八、前期阶段最值得固定的 3 种输出

如果你想让 Claude 前期输出更稳定，建议每次尽量让它产出这 3 种东西。

### 1. 需求摘要

包含：

- 目标
- 用户
- 场景
- MVP
- 风险点

### 2. 方案比较

包含：

- 方案 A / B / C
- 各自 tradeoff
- 推荐方案

### 3. 任务清单

包含：

- 小任务
- 顺序
- 依赖
- 验证方式

这 3 个输出加起来，基本就把“开始写代码之前最重要的脑力工作”做完了。

## 九、前期阶段最实用的对话模板合集

## 1. 需求澄清

```text
先不要写代码。
请帮我把这个需求整理成清晰版本，输出问题定义、目标用户、核心场景、MVP 和风险点。
```

## 2. 边界收敛

```text
请帮我明确这个需求第一版该做什么、不该做什么，并说明这样划分的理由。
```

## 3. 方案比较

```text
请给我 2-3 种实现方案，比较开发速度、复杂度、维护成本和风险，最后给推荐方案。
```

## 4. 目录结构设计

```text
请基于这个需求和技术方案，设计一个适合当前阶段的目录结构。先出方案，不写代码。
```

## 5. 任务拆分

```text
请把这个需求拆成小任务，要求每一步边界清晰、顺序合理、可独立验证。
```

## 6. 进入开发前准备

```text
请根据当前方案，生成这个项目的初始版 CLAUDE.md，并建议最先要有的 skills 和 agents。
```

## 十、前期阶段最容易踩的坑

### 1. 需求还没清楚就让 Claude 写代码

结果往往是：

- 做多了
- 做偏了
- 做得太重了

### 2. 一开始就追求完美架构

结果往往是：

- 目录很漂亮
- 项目很复杂
- 第一版很难落地

### 3. 没拆 MVP

结果往往是：

- Claude 会把很多“应该有”的东西一股脑补齐

### 4. 任务没拆就直接开工

结果往往是：

- 会话很快失控
- 改动范围变大
- review 成本高

---

## 十一、前期工作和开发工作怎么衔接

最推荐的衔接方式是：

1. 先做需求澄清
2. 再做方案比较
3. 再做任务规划
4. 再生成 `CLAUDE.md`
5. 再确定要不要补 skills / agents
6. 最后再进入开发工作流

也就是说：

**前期工作流负责“想清楚”，开发工作流负责“做出来”。**

---

## 十二、一句话总结

如果开发阶段最重要的是“稳定推进”，那么前期阶段最重要的就是：

**让 Claude 帮你把需求讲清楚、方案比清楚、任务拆清楚，再开始写代码。**
