# Claude 老项目前置阶段对话脚本模板

## 文档目的

这份模板适合：

- 接手一个已有项目
- 在老项目上新增需求
- 在老项目上做重构或较大改动

核心目标：

在真正开始改代码之前，让 Claude 先帮你完成：

1. 项目摸底
2. 风险识别
3. 新需求影响分析
4. 方案比较
5. 任务规划
6. `CLAUDE.md` 初稿或补充

---

## 使用原则

老项目最容易出的问题是：

**Claude 没搞懂上下文就开始改。**

所以顺序一定是：

1. 先摸清项目
2. 再判断约束
3. 再分析需求影响
4. 再比较方案
5. 再拆任务
6. 最后才进入实现

---

## 第一轮：项目摸底

直接复制：

```text
先不要改代码。
请帮我快速摸清这个老项目：
1. 技术栈
2. 主要目录
3. 核心模块
4. 启动 / 测试方式
5. 可能的历史包袱或风险点

最后给我一个“接手这个项目的摘要”。
```

这一轮的目标：

- 让 Claude 先理解项目，不要直接进入修改

---

## 第二轮：识别约束和高风险区域

直接复制：

```text
基于刚才的项目分析，
请进一步帮我识别：
1. 哪些模块最核心
2. 哪些目录或逻辑最敏感
3. 哪些地方最容易改出回归
4. 哪些地方应该尽量避免大改
```

这一轮的目标：

- 先知道哪些地方不能乱碰

---

## 第三轮：分析新需求影响范围

直接复制：

```text
这是一个老项目上的新需求。
先不要改代码。

请帮我分析：
1. 这个需求会影响哪些已有模块
2. 哪些地方可能最容易产生回归
3. 现有结构是否适合直接承载这个需求
4. 是扩展现有逻辑更合适，还是单独加一层更合适
5. 推荐的实现路径是什么
```

这一轮的目标：

- 先理解“新需求 + 老结构”的冲突点

---

## 第四轮：方案比较

直接复制：

```text
先不要写代码。
请针对这个老项目上的需求给我 2-3 种实现方案。
请从下面几个维度比较：
1. 改动范围
2. 回归风险
3. 维护成本
4. 实现复杂度
5. 与现有结构的兼容性

最后给我一个最推荐的方案，并说明为什么。
```

这一轮的目标：

- 不只看“能不能做”
- 更要看“在老项目里怎么做代价最小”

---

## 第五轮：任务规划

直接复制：

```text
现在方案基本确定。
先不要写代码。

请把这次改动拆成 6-10 个小任务。
要求：
1. 每个任务边界清晰
2. 顺序合理
3. 标明依赖关系
4. 尽量减少每轮改动范围
5. 每个任务都能独立验证
```

这一轮的目标：

- 降低老项目改动失控风险

---

## 第六轮：生成或补充 `CLAUDE.md`

如果项目还没有 `CLAUDE.md`，直接复制：

```text
请根据前面的分析，为这个老项目生成一个初始版 CLAUDE.md。
重点写：
1. 项目结构
2. 常用命令
3. 高风险模块
4. 改动原则
5. 默认协作方式
```

如果项目已经有 `CLAUDE.md`，直接复制：

```text
请根据这次新需求和项目分析，
给我一份建议补充到 CLAUDE.md 的内容。
重点补充：
1. 风险区域
2. 默认改动原则
3. review 和验证要求
```

这一轮的目标：

- 把“老项目约束”沉淀进长期规则

---

## 第七轮：建议最小 skills / agents 集合

直接复制：

```text
基于这个老项目当前阶段，
请建议我最先建立哪些 skills 和 agents。

要求：
1. 只推荐最小必要集合
2. 优先考虑老项目高风险改动场景
3. 按投入低、收益高排序
```

这一轮的目标：

- 先建立最小可用框架，不要一次搞太重

通常老项目最先值得有的是：

- `review`
- `bugfix`
- `implementer`
- `debugger`
- `reviewer`

---

## 第八轮：进入实现前确认

直接复制：

```text
现在请把前面这些结论整理成一个“进入开发前检查清单”。
要求包含：
1. 项目结构是否理解清楚
2. 高风险区域是否识别清楚
3. 推荐方案是否已确认
4. 任务是否已拆分
5. CLAUDE.md 是否已生成或更新
6. 最小 skills / agents 是否已明确
```

这一轮的目标：

- 确认不是“边想边乱改”

---

## 最后进入开发时的起始话术

当前面都完成后，再发：

```text
现在开始进入实现阶段。
请按我们已经确认的方案和任务顺序推进。
要求：
1. 优先最小改动
2. 每次只推进一个小任务
3. 改完说明修改点、验证结果和残余风险
4. 如果发现影响范围比预期大，先停下来告诉我
```

---

## 一句话总结

老项目里，Claude 最好的用法不是直接帮你改，而是先帮你完成：

**项目摸底 -> 风险识别 -> 需求影响分析 -> 方案比较 -> 任务规划 -> 规则补充**
