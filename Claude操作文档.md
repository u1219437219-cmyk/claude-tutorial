# Claude 项目协作操作文档

## 目标

这份文档用于整理 Claude 在两类场景中的推荐用法：

- 新项目或项目初期如何协作
- 空白项目从 0 到 1 如何启动

核心原则不是“装了多少插件和技能”，而是先给 Claude 一个稳定的项目协作框架。否则能力很多，实际还是会乱用、漏用、重复用。

---

## 一、项目初期的总原则

### 核心思路

优先建立这条主线：

1. 先让 Claude 理解项目
2. 再固定规则
3. 再沉淀高频流程
4. 最后才做自动化

### 推荐分层

- `CLAUDE.md`
  作用：项目长期规则
  什么时候用：一直生效

- `skills`
  作用：高频任务模板
  什么时候用：反复出现的任务

- `agents`
  作用：复杂任务分工
  什么时候用：一个人脑子不够清晰，或者想拆角色

- `plugins`
  作用：整套工作流
  什么时候用：某类流程已经成熟，想系统化复用

- `hooks`
  作用：自动校验或自动收尾
  什么时候用：总忘记做某个固定动作

一句话：

- 规则写进 `CLAUDE.md`
- 经验沉淀成 `skill`
- 分工交给 `agent`
- 成熟流程打包成 `plugin`
- 重复动作交给 `hook`

## 二、新项目或项目初期怎么用

目标不是一上来就自动化拉满，而是先把基础协作面搭起来。

### 先做这 4 件事

#### 1. 写一个最小版 `CLAUDE.md`

建议内容：

- 项目是干什么的
- 技术栈
- 目录结构
- 常用命令
- 代码规范
- 哪些文件不能乱动
- 希望 Claude 默认怎么工作

可直接对 Claude 这样说：

```text
请根据当前仓库生成一个最小可用的 CLAUDE.md。
要求包含：项目目标、技术栈、目录约定、开发命令、测试命令、改代码时的注意事项。
先分析仓库，再生成。
```

#### 2. 先定义 2-3 个最常用 skill / 工作流

不要一开始做太多，优先高频场景：

- 代码审查
- 新功能实现
- bug 排查
- 文档生成

可直接这样说：

```text
基于这个项目，帮我设计 3 个最值得先有的 skills。
要求优先覆盖：功能开发、bug 修复、代码 review。
每个都说明用途、触发场景和建议结构。
```

#### 3. 给项目配 1-2 个 agent 就够

通常最先有价值的是：

- `implementation-agent`
- `reviewer`
- `debugger`

可直接这样说：

```text
请为这个项目设计两个最实用的 subagents。
一个负责实现功能，一个负责代码审查。
告诉我它们各自的职责边界和建议放进 .claude/agents/ 的定义方式。
```

#### 4. 把默认协作方式固定下来

推荐默认节奏：

- 小任务：直接做
- 中任务：先读代码再改
- 大任务：先计划再改
- 高风险任务：先分析 + checkpoint
- 改完：说明改动 + 风险 + 验证结果

可以写进 `CLAUDE.md`，也可以直接对 Claude 说：

```text
以后在这个项目里，默认按这个方式协作：
小改动直接做；
多文件修改先分析相关文件再动手；
复杂需求先给计划；
改完要说明改了什么、有没有风险、做了哪些验证。
```

## 三、老项目改造怎么用

老项目最容易出的问题是：Claude 没搞懂上下文就开始改。

推荐顺序：

1. 先让 Claude 建立项目认知
2. 再抽取规则到 `CLAUDE.md`
3. 再定义高频工作流
4. 最后才做更强自动化

### 推荐对话顺序

#### 第一轮：摸清项目

```text
先不要改代码。
请帮我快速摸清这个老项目：
1. 技术栈
2. 主要目录
3. 核心模块
4. 启动/测试方式
5. 明显的历史包袱或风险点
最后给我一个接手这个项目的摘要。
```

#### 第二轮：生成 `CLAUDE.md` 初稿

```text
根据你刚才的分析，帮我生成适合这个项目的 CLAUDE.md 初稿。
重点写：项目结构、常用命令、危险目录、改动原则、测试要求。
```

#### 第三轮：建立高频工作流

```text
基于这个老项目，建议我最先建立哪些 skills / agents / hooks。
要求按“投入低、收益高”排序。
```

## 四、做新功能时怎么和 Claude 对话

推荐流程：

1. 先让 Claude 读项目
2. 再让 Claude 出计划
3. 再让它实现
4. 再让 reviewer 类 skill 或 agent 复查

### 示例对话

#### 第一步：先分析，不改代码

```text
我们要给这个项目加一个用户通知功能。
先不要改代码。
先分析相关模块、可能要改的文件、风险点，并给出一个实施计划。
```

#### 第二步：按计划实现

```text
按这个计划开始实现。
要求尽量少改文件，保持现有风格，改完后告诉我修改点和验证结果。
```

#### 第三步：做 review

```text
现在请从 code review 的角度重新检查这次改动。
重点看行为回归风险、边界情况和测试缺口。
```

## 五、修老 bug 时怎么和 Claude 对话

对于老项目 bug，推荐固定这个流程：

1. 先分析，不修
2. 再定位最可能根因
3. 再提出修法
4. 最后再改

### 示例对话

#### 第一步：定位根因

```text
这个 bug 出现在订单提交后偶发重复创建。
先不要改代码。
先分析可能根因，按概率排序，并指出你最想先看的文件。
```

#### 第二步：要最小修复方案

```text
现在给我一个最小修复方案，要求尽量少改，优先降低回归风险。
```

#### 第三步：开始修改

```text
可以开始改。
改完后请说明：
1. 改了哪些文件
2. 为什么这样改
3. 还有哪些残余风险
4. 建议补哪些测试
```

## 六、什么时候该真正上 plugin / hook

不是越早越好。

### 先用 plugin 的时机

- 已经反复做同一类流程
- 比如 PR review、文档同步、部署检查
- 希望团队都按同样方式做

### 先用 hook 的时机

- 总忘记某个动作
- 比如改完忘跑测试、忘检查格式、忘输出总结
- 这个动作规则明确，适合自动化

如果流程还没稳定，不要急着上 hooks 和 plugin。先跑通几轮手动协作，再固化。

---

## 七、项目初期最推荐的最小组合

无论新项目还是老项目，都建议先有这套最小组合：

- 一个 `CLAUDE.md`
- 两个 skill
  - `implementation`
  - `review`
- 两个 agent
  - `implementer`
  - `reviewer`
- 一个简单 hook
  - 改完提醒做验证或总结

这套已经足够比“裸聊 Claude”强很多。

## 八、空白新项目从 0 到 1 的推荐顺序

空白项目最好的用法不是让 Claude 直接从 0 一把梭，而是先搭出：

- 项目骨架
- 协作规则
- 初始节奏

Claude 最适合做 4 类事：

- 帮你澄清需求
- 帮你搭项目结构
- 帮你建立规则
- 帮你推进首批实现

### 推荐启动顺序

#### 第 1 步：先定义项目目标

```text
我要从 0 开始做一个新项目。
目标是：做一个给内部团队使用的任务管理 Web 应用。
用户是：产品和研发团队。
核心功能包括：登录、任务创建、任务分配、状态流转、评论。
请先帮我把需求整理成一个简洁清晰的项目说明，不要写代码。
```

#### 第 2 步：拆 MVP

```text
基于刚才这个项目说明，请帮我拆分 MVP。
要求分成：
1. 第一版必须有的功能
2. 可以第二阶段再做的功能
3. 暂时不做的功能
并说明这样拆分的理由。
```

#### 第 3 步：选技术方案

```text
现在请基于这个 MVP，帮我推荐技术方案。
请从开发速度、维护成本、部署难度、团队学习成本四个维度比较。
最后给出一个适合首版落地的建议栈。
```

#### 第 4 步：设计初始目录结构

```text
基于这个项目目标和技术选型，请设计一个适合 MVP 的项目目录结构。
要求：
1. 不要过度设计
2. 结构清晰
3. 便于后续扩展
4. 说明每个主要目录的职责
先出结构方案，不写代码。
```

#### 第 5 步：生成初始版 `CLAUDE.md`

```text
请根据我们刚才确定的项目目标、MVP 和技术方案，生成一个新项目初始版 CLAUDE.md。
要求包含：
1. 项目目标
2. 技术栈
3. 目录约定
4. 开发原则
5. 默认协作方式
6. 改代码时的约束
```

#### 第 6 步：拆第一批开发任务

```text
请基于当前方案，把项目的第一批开发任务拆成 8-12 个小任务。
要求：
1. 每个任务尽量独立
2. 有合理顺序
3. 标注前置依赖
4. 适合逐步实现和验证
```

#### 第 7 步：从骨架开始实现

```text
现在开始实现第一阶段。
先帮我完成项目初始化和基础目录搭建。
要求：
1. 保持最小可运行
2. 不要提前做没必要的抽象
3. 改完告诉我创建了哪些文件以及下一步建议
```

## 九、空白项目里最常见的好对话方式

### 1. 用 Claude 做需求澄清

```text
我现在只有一个模糊想法，请帮我把它澄清成可执行需求。
目标用户是……
核心问题是……
希望你输出：用户场景、核心功能、MVP 边界。
```

### 2. 用 Claude 做架构收敛

```text
不要先追求完美架构。
请帮我给这个项目设计一个适合 0 到 1 的最小结构，并解释为什么这样设计。
```

### 3. 用 Claude 做任务拆分

```text
请把这个功能拆成足够小的实现步骤，每一步都能独立验证。
```

### 4. 用 Claude 做首版实现

```text
现在只做 MVP，不做未来扩展预留。
请按最小可运行原则实现。
```

### 5. 用 Claude 做规则沉淀

```text
把我们刚刚确定的约定整理进 CLAUDE.md，避免后续协作漂移。
```

## 十、空白项目里最容易踩的坑

### 1. 一开始就让 Claude 生成太多

比如：

- 一次生成完整后台
- 一次生成全量前端
- 一次搞完整权限系统

常见结果：

- 结构虚胖
- 抽象过度
- 细节不稳定

### 2. 没先明确 MVP

Claude 会倾向于补全很多“看起来应该有”的东西，所以要不断提醒：

```text
只做第一版必须有的，不做增强功能。
```

### 3. 没建立规则

空白项目最适合一开始就把规则写下来，否则几天后项目风格就会漂。

### 4. 过早上 hooks / plugin

在流程还没稳定前，先别急着自动化。先把人和 Claude 的基本协作跑顺。

## 十一、空白项目推荐开场话术

下面这段几乎可以直接复制：

```text
我要从 0 开始做一个新项目。
先不要写代码，先帮我完成这几件事：
1. 把我的想法整理成清晰的项目目标
2. 拆出 MVP 范围
3. 推荐合适的技术方案
4. 设计最小可扩展的目录结构
5. 生成项目初始版 CLAUDE.md
6. 把第一阶段开发任务拆出来

整个过程请遵循：
- 优先可落地，不要过度设计
- 优先 MVP，不提前做复杂扩展
- 每一步先解释思路，再继续下一步
```

---

## 十二、当准备正式开始编码时

可以切成这种节奏：

```text
现在开始做第一阶段实现。
先完成基础项目初始化。
要求：
1. 保持结构简单
2. 每次只推进一个小里程碑
3. 每完成一步，告诉我改了什么、下一步是什么
```

然后每轮都只让 Claude 推进一小步。

## 十三、常用对话模板速查

### 接手项目

```text
先不要改代码，帮我快速理解这个项目，并生成一份适合 Claude 协作的项目摘要。
```

### 做新功能

```text
先分析需求会影响哪些模块，再给计划，等我确认后再改。
```

### 修 bug

```text
先定位根因，不要急着修。给我最可能的原因、证据和最小修复方案。
```

### 做 review

```text
请从 reviewer 角度检查这次改动，重点看回归风险、测试缺口和可维护性。
```

### 沉淀规则

```text
把这次我们形成的协作规则整理进 CLAUDE.md / skill 草稿里。
```

## 总结

无论是新项目、老项目，还是空白项目，从 Claude 获得稳定收益的关键都不是“堆能力”，而是先建立一套可持续的协作框架：

1. 让 Claude 理解项目
2. 用 `CLAUDE.md` 固化规则
3. 用 `skills` 固化高频任务
4. 用 `agents` 做任务分工
5. 等流程成熟后，再上 `hooks` 和 `plugins`

空白项目里，Claude 最强的用法不是“一口气写完整项目”，而是帮助你把模糊想法逐步收敛成可执行的项目结构、规则和首批实现。

## 十四、个性化工作流模板

这一节适合日常开发直接复制使用。默认风格是：

- 先理解，再动手
- 先收敛范围，再实现
- 优先小步推进
- 每一步都要有结果说明

### 1. 接手一个已有项目

```text
先不要改代码。
请先帮我快速接手这个项目，输出：
1. 技术栈
2. 目录结构
3. 核心模块
4. 启动和测试方式
5. 风险点和历史包袱

最后给我一份“我接下来应该先做什么”的建议列表。
```

### 2. 做一个新功能

```text
我们现在要做一个新功能。
先不要直接改代码。

请先完成这几步：
1. 找出相关模块和文件
2. 分析实现思路和风险点
3. 给一个尽量小步推进的计划

等我确认后再开始实现。
实现时要求：
- 尽量少改文件
- 保持现有代码风格
- 不做不必要的抽象

改完后请说明：
1. 改了哪些文件
2. 为什么这样改
3. 做了哪些验证
4. 还有哪些残余风险
```

### 3. 修一个 bug

```text
先不要改代码。
帮我定位这个 bug：
[这里写现象]

请按下面格式输出：
1. 最可能的根因
2. 相关文件
3. 建议先验证什么
4. 最小修复方案

等我确认后再开始改。
改动时要求优先降低回归风险，尽量少改。
```

### 4. 做一次代码 review

```text
请从 reviewer 的角度检查这次改动。
重点看：
1. 行为回归风险
2. 边界情况
3. 测试缺口
4. 可维护性问题

先列问题，再给建议。
如果没有明显问题，也请告诉我还存在哪些残余风险。
```

### 5. 沉淀规则到 `CLAUDE.md`

```text
把这次我们形成的协作规则整理进 CLAUDE.md 草稿。
重点补充：
1. 项目约定
2. 默认协作方式
3. 常用命令
4. 改代码时的边界和注意事项
5. 验证要求
```

### 6. 空白项目从 0 到 1

```text
我要从 0 开始做一个新项目。
先不要写代码，先帮我依次完成：
1. 整理项目目标
2. 拆 MVP
3. 推荐技术方案
4. 设计最小目录结构
5. 生成初始版 CLAUDE.md
6. 拆第一批开发任务

要求：
- 优先可落地
- 不要过度设计
- 每一步先解释思路，再进入下一步
```

### 7. 默认协作口径

如果你想在一个项目里长期保持稳定节奏，可以直接对 Claude 说：

```text
以后在这个项目里默认按这个方式协作：
- 小改动直接做
- 多文件修改先分析再动手
- 复杂需求先给计划
- 高风险任务先分析并保守推进
- 改完要说明改动、验证结果和残余风险
```
