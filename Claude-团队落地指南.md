# Claude 团队落地指南

## 文档目的

这份文档补的是团队视角的问题：

- 如果要在团队里推广 Claude，应该怎么落地
- 如何避免每个人各用各的，最后变乱
- 如何把个人经验变成团队可复用资产

---

## 一、团队落地的核心原则

### 1. 先统一方法，再统一工具

不要一开始就在团队里推广一堆 skill、agent、plugin。

先统一：

- Claude 应该怎么参与工作
- 什么任务适合交给 Claude
- 什么任务不能直接交给 Claude
- 输出结果怎么 review

一句话：

**先统一协作方式，再统一配置。**

### 2. 先试点，再推广

最稳的方式：

1. 先找 1-2 个试点项目
2. 跑通最小协作框架
3. 形成规则和模板
4. 再推广到更多项目

### 3. 先解决高频问题

团队最先值得统一的通常是：

- 新功能怎么推进
- bug 怎么排查
- review 怎么做
- 规则怎么沉淀

---

## 二、团队落地的推荐阶段

### 阶段 1：试点阶段

目标：

- 先证明 Claude 真有价值

推荐动作：

1. 选 1-2 个试点项目
2. 给试点项目加 `CLAUDE.md`
3. 建最小 skills / agents
4. 先只覆盖高频任务

推荐最小集合：

- `CLAUDE.md`
- `review` skill
- `bugfix` skill
- `implementer` agent
- `reviewer` agent

### 阶段 2：固化阶段

目标：

- 把试点经验变成团队统一规则

推荐动作：

1. 把有效规则补进 `CLAUDE.md`
2. 沉淀团队通用 skill
3. 统一 agent 职责命名
4. 整理常见对话模板

### 阶段 3：推广阶段

目标：

- 复制到更多项目和成员

推荐动作：

1. 做项目模板包
2. 做版本选择指南
3. 建团队总目录索引
4. 统一外部能力引入流程

### 阶段 4：工程化阶段

目标：

- 让团队使用更稳定、更低成本

推荐动作：

1. 补简单 hooks
2. 引入只读型 MCP
3. 沉淀 plugin 级工作流
4. 明确维护责任人

---

## 三、团队里最值得先统一的内容

### 1. `CLAUDE.md` 写法

建议统一：

- 栏目结构
- 默认协作方式
- 改动前后要求
- review 和验证要求

### 2. 高风险任务默认流程

团队里最好统一：

- 复杂需求先计划
- 高风险改动先分析
- 大改动前用 checkpoint / worktree
- 改完必须 review / 自检

### 3. 常用技能和角色命名

比如统一这些：

- `review`
- `bugfix`
- `feature`
- `refactor`
- `implementer`
- `reviewer`
- `debugger`

如果前后端要区分，也统一成：

- `frontend-review`
- `backend-review`
- `frontend-implementer`
- `backend-implementer`

### 4. 外部能力引入流程

建议统一：

1. 先评估
2. 再小范围试
3. 再项目化改造
4. 最后再正式并入

---

## 四、团队里最容易踩的坑

### 1. 每个人自己维护一套

结果：

- 名字不统一
- 风格不统一
- 经验无法复用

### 2. 一开始就上很重的配置

结果：

- 学习成本高
- 团队接受度低
- 维护成本高

### 3. 只教大家“怎么装”，不教“怎么用”

结果：

- 文件都有
- 实际不会用

### 4. 不指定维护人

结果：

- `CLAUDE.md` 过期
- skills 过期
- 文档过期

---

## 五、团队角色建议

### 1. 试点负责人

负责：

- 推进试点项目
- 收集问题
- 形成第一版规则

### 2. 资料维护人

负责：

- 维护团队模板
- 更新 `CLAUDE.md` 基线
- 更新常用 skill / agent 模板

### 3. 项目负责人

负责：

- 判断项目适合通用版、前端版、后端版还是全栈版
- 决定项目级规则如何落地

---

## 六、一句话总结

Claude 在团队里最好的落地方式，不是让每个人自己研究，而是：

**先试点，先统一协作方式，再逐步沉淀成团队模板和项目规范。**
