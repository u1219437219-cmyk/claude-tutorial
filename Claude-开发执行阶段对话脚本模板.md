# Claude 开发执行阶段对话脚本模板

## 文档目的

这份模板补的是流程后半段：

- 开始实现
- review / 自检
- bug 分析与修复
- 联调 / 验证
- 进入下一轮迭代

对应你前面的流程图后半段：

```text
开始实现 -> review / 自检 -> 是否有问题
有问题 -> bug 分析与修复 -> review / 自检
无问题 -> 联调 / 验证 -> 是否稳定
不稳定 -> bug 分析与修复
稳定 -> 进入下一轮迭代
```

目标是给你一套“开发执行阶段怎么和 Claude 对话”的可复制脚本。

---

## 一、总原则

开发执行阶段最重要的不是“让 Claude 一口气做完”，而是：

1. 一次只推进一个小任务
2. 每轮都做 review / 自检
3. 发现问题先定位根因，再修
4. 联调后再决定是否进入下一轮

一句话：

**小步实现，快速复查，问题闭环。**

---

## 二、开始实现

这一轮是在前期阶段已经完成后才使用。

### 适用场景

- 需求已经澄清
- MVP 已确定
- 方案已选定
- 任务已拆分
- `CLAUDE.md` 和最小 skills / agents 已准备

### 对话模板

```text
现在开始进入实现阶段。
请按我们已经确认的方案和任务顺序推进。

要求：
1. 每次只推进一个小任务
2. 尽量少改文件
3. 保持现有结构和风格
4. 如果发现影响范围比预期大，先停下来告诉我

改完后请说明：
1. 改了哪些文件
2. 实现了什么
3. 做了哪些验证
4. 还有哪些残余风险
```

### 如果你想更稳一点

可以补一句：

```text
先完成当前这个小任务，不要顺手扩展到下一个任务。
```

---

## 三、实现后的 review / 自检

这是每轮实现后都建议做的一步。

### 目标

- 看有没有回归风险
- 看有没有边界情况遗漏
- 看测试和验证是否不足
- 看复杂度有没有不必要升高

### 通用 review 对话模板

```text
请对刚才这轮改动做一次 review / 自检。
重点看：
1. 行为回归风险
2. 边界情况
3. 测试缺口
4. 可维护性问题

请先列 findings，再给总结。
如果没有明显问题，也请告诉我还存在哪些残余风险。
```

### 如果是前端任务

```text
请按前端 review 的口径检查刚才这轮改动。
重点看：
1. UI 回归风险
2. 交互一致性
3. 状态管理是否复杂化
4. 样式和响应式是否受影响
5. 测试和手动验证是否充分
```

### 如果是后端任务

```text
请按后端 review 的口径检查刚才这轮改动。
重点看：
1. 接口契约是否被破坏
2. 数据一致性风险
3. 错误处理和日志是否充分
4. 并发 / 事务 / 缓存 / 消息是否有隐患
5. 测试覆盖是否足够
```

---

## 四、发现问题后的 bug 分析

如果 review / 自检发现问题，不要马上让 Claude 直接重改，先做 bug 分析。

### 目标

- 先定位根因
- 再给最小修复方案
- 避免反复来回乱改

### 通用 bug 分析模板

```text
刚才这轮改动里发现了问题。
先不要直接改代码。

请先帮我分析：
1. 最可能的根因
2. 涉及哪些文件或模块
3. 建议先验证什么
4. 最小修复方案是什么

请优先修根因，不要只做表面补丁。
```

### 如果是前端问题

```text
这轮前端改动里发现了问题。
先不要直接改代码。

请先分析：
1. 最可能的根因
2. 相关页面、组件、hooks、样式或状态逻辑
3. 建议先验证什么
4. 最小修复方案
```

### 如果是后端问题

```text
这轮后端改动里发现了问题。
先不要直接改代码。

请先分析：
1. 最可能根因
2. 涉及的接口、服务、数据层、任务或日志位置
3. 建议先验证的数据或日志
4. 最小修复方案
```

---

## 五、执行 bug 修复

当根因已经比较明确，再进入修复。

### 通用修复模板

```text
现在按你刚才给出的最小修复方案开始修改。

要求：
1. 优先最小改动
2. 不做无关重构
3. 修完后说明：
   - 改了哪些文件
   - 为什么这样改
   - 做了哪些验证
   - 还有哪些残余风险
```

### 修复后的复查模板

```text
请再对这次修复做一次 review / 自检。
重点看：
1. 原问题是否真正闭环
2. 是否引入新的回归风险
3. 验证是否充分
4. 是否还有残余问题
```

---

## 六、联调 / 验证

如果一轮实现或修复之后，没有明显问题，就进入联调 / 验证阶段。

### 目标

- 看功能是不是整体跑通
- 看前后端是不是对齐
- 看关键路径是不是稳定

### 通用联调模板

```text
现在进入联调 / 验证阶段。

请帮我整理：
1. 这轮改动需要验证的关键路径
2. 最推荐的验证顺序
3. 哪些场景必须重点检查
4. 哪些地方最可能还有残余风险
```

### 全栈联调模板

```text
这是一次跨前后端改动。
现在请帮我整理联调 / 验证清单：
1. 前端需要验证什么
2. 后端需要验证什么
3. 前后端联动路径怎么验证
4. 哪些契约或数据联动最值得重点检查
5. 如果出问题，优先从哪边查
```

### 验证结果回收模板

```text
基于当前改动，请帮我总结这轮验证结果。

输出：
1. 已验证通过的部分
2. 仍不稳定或不确定的部分
3. 建议下一步补什么修复或验证
```

---

## 七、判断是否稳定

这一轮的判断标准通常是：

- 核心功能跑通
- 主要回归风险已检查
- 关键路径可验证
- 没有明显新的 blocker

### 让 Claude 帮你判断是否进入下一轮

```text
请基于当前实现、review、bugfix 和验证结果，
帮我判断这轮是否已经足够稳定，可以进入下一轮迭代。

请输出：
1. 可以进入下一轮的理由
2. 还不建议进入下一轮的原因
3. 如果现在进入下一轮，最大的风险是什么
```

---

## 八、进入下一轮迭代

如果这一轮已经稳定，就进入下一轮。

### 对话模板

```text
这一轮先收口。

请帮我做一个本轮迭代总结，输出：
1. 本轮完成了什么
2. 还剩哪些待办
3. 哪些规则应该补进 CLAUDE.md
4. 哪些高频问题值得沉淀成 skill / agent / hook
5. 下一轮最建议先做什么
```

### 为什么这一步重要

因为它能把“本轮经验”沉淀下来，而不是每轮都重来。

---

## 九、推荐的开发执行闭环

最推荐你每个小任务都按这个闭环走：

1. 实现一个小任务
2. 做 review / 自检
3. 如果有问题，先做 bug 分析
4. 再做最小修复
5. 再 review
6. 再联调 / 验证
7. 稳定后进入下一轮

一句话：

**实现 -> 复查 -> 修复 -> 验证 -> 进入下一轮**

---

## 十、最容易踩的坑

### 1. 一轮实现做太多

结果：

- review 很难做
- bug 很难定位
- 风险很难控制

### 2. 有问题时直接让 Claude 重写

结果：

- 根因没搞清
- 代码来回反复改

### 3. 不做联调，只看代码

结果：

- 前后端对不齐
- 页面和接口联动问题晚发现

### 4. 每轮结束不总结

结果：

- 下轮又重新解释
- 经验不会沉淀

---

## 十一、一句话总结

开发执行阶段，Claude 最好的用法不是“让它一路往前写”，而是：

**每推进一小步，就让它完成一次“实现 -> review -> bugfix -> 验证 -> 总结”的闭环。**
