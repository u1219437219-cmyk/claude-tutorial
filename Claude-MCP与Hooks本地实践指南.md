# Claude MCP 与 Hooks 本地实践指南

## 文档目的

前面的资料已经讲了：

- MCP 是什么
- hooks 是什么
- 它们为什么重要

这份文档补的是更落地的一层：

**在本地项目里，MCP 和 hooks 应该什么时候上、怎么上、先上什么最稳。**

---

## 一、先给结论

### MCP 先从“只读型”开始

最推荐先接：

- GitHub 只读查询
- 文档 / 知识库查询
- 日志查询

不建议一开始就接：

- 高权限数据库写操作
- 生产环境变更能力
- 会直接影响外部系统状态的能力

### hooks 先从“提醒型”开始

最推荐先上：

- 改完提醒说明验证
- 改完提醒说明风险
- 改完提醒总结变更

不建议一开始就上：

- 复杂自动决策
- 频繁拦截型 hook
- 规则还没稳定时的大量自动化

---

## 二、什么时候该上 MCP

只有当你发现 Claude 在项目里反复需要这些外部信息时，MCP 才值得接：

- 需要读 GitHub PR / issue / 评论
- 需要查数据库结构或数据
- 需要查日志平台
- 需要查 Notion / 文档系统
- 需要接入某个业务后台系统

### 不该上 MCP 的情况

- 当前项目主要还是本地代码理解
- 还没形成稳定工作流
- 只是“感觉以后可能会用”

一句话：

**外部信息成为刚需时再上，不要为了完整而接。**

---

## 三、什么时候该上 hooks

当你发现“某件事总是忘记做”，就说明 hook 值得引入。

最典型的信号：

- 改完总忘说明验证结果
- 改完总忘说明残余风险
- 做完任务总忘总结本轮变化
- 提交前总忘做一项固定检查

### 不该急着上 hooks 的情况

- 当前规则还在变
- 你还没确定团队真正想统一什么
- 当前 hook 会显著打断正常协作

一句话：

**流程稳定以后再自动化。**

---

## 四、MCP 最推荐的本地引入顺序

### 第 1 阶段：只读型 MCP

优先级最高，风险最低。

适合：

- 查 GitHub issue / PR
- 查文档
- 查日志

### 第 2 阶段：分析辅助型 MCP

例如：

- 数据库 schema 查询
- 只读业务数据分析

### 第 3 阶段：有副作用的 MCP

例如：

- 创建 issue
- 修改外部文档
- 改数据库内容

这类一定要谨慎，通常不建议项目初期就开。

---

## 五、Hooks 最推荐的本地引入顺序

### 第 1 阶段：提醒型 hooks

例如：

- 改完提醒总结
- 改完提醒验证

### 第 2 阶段：轻检查型 hooks

例如：

- 简单校验输出格式
- 提醒不要漏关键步骤

### 第 3 阶段：强约束型 hooks

例如：

- 拦截某些高风险动作
- 强制执行某些流程

这类只有当规则极稳定时才值得上。

---

## 六、项目里最实用的 hooks 起步模板

这是最推荐的第一批 hook 思路：

### 1. 改完提醒型

目的：

- 提醒 Claude 说明：
  - 改了什么
  - 做了哪些验证
  - 残余风险是什么

### 2. 高风险任务提醒型

目的：

- 当涉及多文件或关键目录时，提醒：
  - 先分析影响范围
  - 先说明风险

### 3. 结束总结型

目的：

- 在一轮任务结束时，提醒总结：
  - 本轮完成什么
  - 下一步做什么

---

## 七、项目里最实用的 MCP 起步模板

### 1. GitHub 只读型

适合：

- 查 PR
- 查评论
- 查 issue
- 看变更上下文

### 2. 文档 / 知识库只读型

适合：

- 查产品文档
- 查规范
- 查历史决策

### 3. 日志 / 观测只读型

适合：

- 辅助 bug 排查
- 查错误上下文

---

## 八、MCP 和 hooks 在项目里的最佳分工

你可以这样记：

- `MCP`：让 Claude 看见外部世界
- `hooks`：让 Claude 在项目流程里更守纪律

也就是说：

- MCP 管“看得到什么”
- hooks 管“该怎么做”

---

## 九、最推荐的实际引入顺序

如果你要给一个真实项目逐步升级，建议这样：

1. 先有 `CLAUDE.md`
2. 再有 skills / agents
3. 再加一个最简单的提醒型 hook
4. 再接一个只读型 MCP
5. 流程稳定后，再考虑更复杂的 plugin / MCP / hooks

---

## 十、一句话总结

如果你要让 Claude 更强，最稳的做法不是一口气把 MCP 和 hooks 都上满，而是：

**先用 hooks 补纪律，再用 MCP 补视野，而且两者都从低风险版本开始。**
