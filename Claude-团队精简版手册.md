# Claude 团队精简版手册

## 1. Claude 协作原则

### 核心原则

- 先理解项目，再动手
- 先固定规则，再沉淀流程
- 优先小步推进，不要一次改太多
- 优先最小修改，不做无关重构
- 改完必须说明修改点、验证结果和残余风险

### 五层分工

- `CLAUDE.md`：项目长期规则
- `skill`：高频任务模板
- `agent`：复杂任务分工
- `plugin`：整套工作流
- `hook`：自动校验和收尾

### 推荐最小配置

- 一个 `CLAUDE.md`
- 两个 skill：`implementation`、`review`
- 两个 agent：`implementer`、`reviewer`
- 一个简单 hook：提醒验证或总结

---

## 2. 项目启动标准流程

### 新项目或项目初期

1. 让 Claude 理解项目
2. 生成最小版 `CLAUDE.md`
3. 设计 2-3 个高频 skill
4. 设计 1-2 个 agent
5. 固定默认协作方式

### 老项目改造

1. 先摸清技术栈、目录、核心模块、命令、风险点
2. 再生成 `CLAUDE.md`
3. 再补 skills / agents / hooks
4. 流程稳定后再做 plugin / hook 自动化

### 空白项目从 0 到 1

1. 整理项目目标
2. 拆 MVP
3. 选技术方案
4. 设计目录结构
5. 生成 `CLAUDE.md`
6. 拆第一批任务
7. 从最小骨架开始实现

## 3. 高频对话模板

### 接手项目

```text
先不要改代码，帮我快速理解这个项目，并生成一份适合 Claude 协作的项目摘要。
```

### 做新功能

```text
先分析需求会影响哪些模块，再给计划，等我确认后再改。
```

### 修 bug

```text
先定位根因，不要急着修。给我最可能的原因、证据和最小修复方案。
```

### 做 review

```text
请从 reviewer 角度检查这次改动，重点看回归风险、测试缺口和可维护性。
```

### 沉淀规则

```text
把这次我们形成的协作规则整理进 CLAUDE.md / skill 草稿里。
```

### 默认协作口径

```text
以后在这个项目里默认按这个方式协作：
- 小改动直接做
- 多文件修改先分析再动手
- 复杂需求先给计划
- 高风险任务先分析并保守推进
- 改完要说明改动、验证结果和残余风险
```

## 4. CLAUDE.md 最小规范

项目里的 `CLAUDE.md` 建议至少包含：

- 项目目标
- 技术栈
- 目录约定
- 常用命令
- 默认协作方式
- 开发原则
- 修改前要求
- 修改后要求
- 测试与验证要求
- 风险控制要求

### 可直接使用的初始版本

```md
# 项目协作规范

## 项目目标
- 这个项目的核心目标是：
- 当前阶段最重要的任务是：

## 技术栈
- 主要语言：
- 主要框架：
- 包管理工具：
- 测试工具：

## 目录约定
- `src/`：核心业务代码
- `tests/`：测试代码
- `docs/`：项目文档
- `scripts/`：辅助脚本

## 常用命令
- 开发启动：
- 测试：
- 构建：
- lint：
- format：

## 默认协作方式
- 小改动可以直接做
- 多文件修改先分析相关文件和影响范围
- 复杂需求先给计划，再开始改
- 高风险任务先说明风险和保守方案
- 修改完成后必须说明改动、验证结果和残余风险

## 开发原则
- 优先小步推进
- 优先最小可行修改
- 尽量少改文件，保持现有风格
- 不做无关重构

## 修改后要求
- 说明修改了哪些文件
- 说明为什么这样改
- 说明做了哪些验证
- 说明还存在哪些残余风险
```

## 5. 前端 / 后端补充注意点

### 前端项目

- 改页面和组件前，先阅读现有结构和样式模式
- 优先复用现有组件和工具函数
- 样式和交互改动要说明影响范围
- 改完要说明界面行为、边界情况和验证结果

### 后端项目

- 改接口和数据逻辑前先分析影响范围
- 优先保持现有契约和兼容性
- 涉及数据库、缓存、消息时要说明一致性风险
- 改完要说明验证结果、兼容性影响和残余风险

---

## 6. 最后建议

- 不要一开始把所有插件和技能都用上
- 不要让 Claude 一口气生成完整项目
- 先把协作规则跑顺，再上自动化
- 让 Claude 先分析，再计划，再实现，再 review
