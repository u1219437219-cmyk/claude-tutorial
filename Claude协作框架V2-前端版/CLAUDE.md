## 目录约定

- `src/pages/` 或页面目录：页面级代码
- `src/components/`：组件
- `src/hooks/`：复用逻辑
- `src/styles/`：样式
- `src/services/` 或接口目录：请求逻辑
- `tests/`：测试代码

如果组件、页面、样式边界不明确，先阅读现有结构再修改，不要随意新建层级。

## 默认协作方式

- 改页面和组件前先阅读现有结构和样式模式
- 优先复用现有组件、hooks 和工具函数
- 多文件修改先分析影响的页面、组件、状态和接口
- 复杂需求先给计划，再开始改
- 改完必须说明用户可见变化、验证结果和回归风险

## 开发原则

- 优先小步推进
- 优先保持视觉和交互一致性
- 不随意新增复杂抽象
- 不轻易破坏现有布局、响应式和组件边界
- 如果状态逻辑可能变复杂，先说明方案和风险

## 修改后要求

- 说明修改了哪些页面、组件、状态或样式
- 说明用户可见变化是什么
- 说明做了哪些验证
- 说明是否存在 UI / 交互 / 响应式回归风险

## 测试与验证

- 能跑测试时优先跑最相关的测试
- 如果没有测试，至少说明建议的手动验证路径
- 对 UI 改动要说明主要交互场景如何验证

## 代码审查口径

默认重点关注：

- UI 回归风险
- 交互一致性
- 组件边界是否清晰
- 状态管理是否复杂化
- 样式和响应式是否受影响
- 测试缺口
