```md

- 当前阶段最重要的任务是：
- 当前优先级高于长期优化，先保证可落地和稳定推进。

- 主要语言：
- 主要框架：

## 目录约定

- `src/` 或主业务目录：核心业务代码
- `tests/`：测试代码
- `docs/`：项目文档
- `scripts/`：辅助脚本
- 其他关键目录请根据项目实际情况补充说明

如果目录职责不明确，先阅读相关代码再决定修改位置，不要凭感觉新建结构。

## 常用命令

- 开发启动：
- 测试：
- 构建：
- lint：
- format：

如果不确定命令，先查项目配置，不要猜。

## 默认协作方式

- 小改动可以直接做
- 多文件修改先分析相关文件和影响范围
- 复杂需求先给计划，再开始改
- 高风险任务先说明风险和保守方案
- 修改完成后必须说明改动、验证结果和残余风险

## 开发原则

- 优先小步推进，不要一次改太多
- 优先最小可行修改，不做无关重构
- 尽量少改文件，保持现有结构和风格
- 优先复用现有模式，不随意引入新抽象
- 如果发现需求牵涉范围过大，要先提示，再决定是否继续

## 修改前要求

- 先理解相关模块和调用链
- 不清楚的地方先分析，不要直接写代码
- 涉及多个模块时，先说明影响范围
- 涉及高风险逻辑时，优先给最小改动方案

## 修改后要求

- 说明修改了哪些文件
- 说明为什么这样改
- 说明做了哪些验证
- 说明还存在哪些残余风险
- 如未运行测试或验证，必须明确说明

## 测试与验证

- 能跑相关测试时，优先跑最相关的测试
- 如果没有自动化测试，也要说明建议的手动验证方式
- 不要声称“已经修好”而不说明验证依据

## 风险控制

- 不要随意改动不相关模块
- 不要引入与当前任务无关的大规模重构
- 不要破坏现有接口、行为或目录约定，除非明确说明并得到确认
- 涉及删除、迁移、批量修改时，先说明风险

## 代码审查口径

在 review 或自检时，默认重点关注：

- 行为回归风险
- 边界情况
- 测试缺口
- 可维护性问题
- 不必要复杂度

## 适合直接使用的对话风格

在这个项目里，优先使用以下协作方式：

1. 新功能
   - 先分析模块和风险
   - 再给计划
   - 确认后再实现

2. 修 bug
   - 先定位根因
   - 再给最小修复方案
   - 确认后再修改

3. 做 review
   - 先列问题
   - 再给建议
   - 明确残余风险或测试缺口

## 可逐步补充的内容

后续可以根据项目成熟度逐步补充：

- 更具体的目录说明
- 业务模块说明
- 数据或接口约定
- 前端组件约定或后端接口约定
- 团队自己的 skill / agent / hook 使用规则
```

## 使用建议

推荐你后续这样落地：

1. 先复制这份初稿作为项目的 `CLAUDE.md`
2. 填上项目目标、技术栈、命令和目录约定
3. 跑几轮真实协作后，再把高频规则补进去
4. 等流程稳定后，再补 skill / agent / hook 规范
