# Claude 外部 Skill / Plugin / MCP 引入评估清单模板

## 文档目的

这份模板用于评估你在网上发现的外部能力是否值得引入项目，包括：

- skill
- agent
- plugin
- MCP
- hook

目标不是“看到好东西就装”，而是先评估、再试用、再固化。

---

## 一、先记住一条总原则

**外部能力先当候选组件，不要直接当现成真理。**

最稳的引入顺序：

1. 先判断值不值得引入
2. 再小范围试用
3. 再项目化改造
4. 最后再正式并入主项目

## 三、价值评估

先判断它有没有真实引入价值。

### 评估问题

#### 1. 它解决的是不是高频问题？

- 这个问题在我们的项目里常出现吗？
- 这个问题是否反复需要手工描述？

#### 2. 它能不能明显减少重复劳动？

- 能减少重复提示词吗？
- 能减少重复判断或重复检查吗？

#### 3. 它和当前项目是否强相关？

- 是项目真实会用到的吗？
- 还是“看起来很强，但目前没场景”？

- 说明：

### 结论
- [ ] 值得继续评估

## 四、兼容性评估

这一步是最容易被忽略的。

### 要检查的点

#### 1. 是否和现有 `CLAUDE.md` 冲突

- 协作方式是否相反？
- 风格是否明显不一致？

#### 2. 是否和现有 skill 冲突

- 名字是否冲突？
- 职责是否重复？

#### 3. 是否和现有 agent 冲突

- 是否已有相同角色？
- 是否会让分工更乱？

#### 4. 是否和现有 plugin / MCP / hook 冲突

- 是否重复接了同一类能力？
- 是否会放大系统复杂度？

## 五、风险评估

不同类型的外部能力，风险不同。

### skill / agent 风险

- 输出风格不贴项目
- 角色边界不清
- 容易和现有流程重叠

### plugin 风险

- 一次带入太多东西
- 命名冲突
- 工作流过重

### MCP 风险

- 权限太大
- 接入生产系统风险
- 数据读写边界不清

### hook 风险

- 自动化过度
- 频繁打断
- 规则不稳定时难维护

## 六、试用计划

不要跳过这一步。

### 原则

- 先小范围试
- 不直接全量进主项目
- 不一开始就覆盖现有能力

### 试用方式建议

#### 如果是 skill

- 先只在测试项目或单一任务里试
- 看输出结构和口径是否合适

#### 如果是 agent

- 先只引入一个角色明确的 agent
- 看是否真的能让分工更清楚

#### 如果是 plugin

- 先拆开看内部结构
- 先只借其中最有价值的 1-2 个部分

#### 如果是 MCP

- 先从只读能力试
- 不直接给高权限写操作

#### 如果是 hook

- 先做提醒型 hook
- 不要一开始做复杂自动决策

## 七、项目化改造清单

外部能力通常都不能原样照搬。

### skill / agent 改造重点

- 改命名，避免冲突
- 改描述，贴合项目场景
- 改输出格式，贴合你的工作习惯
- 改职责边界，避免重复

### plugin 改造重点

- 先决定整包用还是拆组件用
- 明确保留哪些 command / skill / agent / hook / mcp
- 去掉和项目无关的部分

### MCP 改造重点

- 明确读写权限
- 明确作用范围
- 只接真正需要的系统

## 十、一句话总结

以后你在网上发现新的 skill、plugin、MCP，不要第一反应是“装上试试”，而应该是：

**先评估值不值，再试小范围，再改成自己的版本，最后再正式并入项目。**
