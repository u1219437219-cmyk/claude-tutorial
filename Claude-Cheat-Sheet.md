# Claude Cheat Sheet

## 一句话原则

先理解项目，再固定规则，再沉淀高频流程，最后做自动化。

---

## 最小配置

- 一个 `CLAUDE.md`
- 两个 skill：`implementation`、`review`
- 两个 agent：`implementer`、`reviewer`
- 一个简单 hook：提醒验证或总结

---

## 五层分工

- `CLAUDE.md`：长期规则
- `skill`：高频任务模板
- `agent`：复杂任务分工
- `plugin`：整套工作流
- `hook`：自动校验或收尾

---

## 新项目启动顺序

1. 让 Claude 理解项目
2. 生成最小版 `CLAUDE.md`
3. 设计 2-3 个高频 skill
4. 设计 1-2 个 agent
5. 固定默认协作方式

---

## 老项目改造顺序

1. 先摸清技术栈、目录、核心模块、命令、风险点
2. 再生成 `CLAUDE.md`
3. 再补 skills / agents / hooks
4. 流程稳定后再做 plugin / hook 自动化

---

## 空白项目顺序

1. 整理项目目标
2. 拆 MVP
3. 选技术方案
4. 设计目录结构
5. 生成 `CLAUDE.md`
6. 拆第一批任务
7. 从最小骨架开始实现

---

## 最常用对话模板

### 接手项目

```text
先不要改代码，帮我快速理解这个项目，并生成一份适合 Claude 协作的项目摘要。
```

### 做新功能

```text
先分析需求会影响哪些模块，再给计划，等我确认后再改。
```

### 修 bug

```text
先定位根因，不要急着修。给我最可能的原因、证据和最小修复方案。
```

### 做 review

```text
请从 reviewer 角度检查这次改动，重点看回归风险、测试缺口和可维护性。
```

### 沉淀规则

```text
把这次我们形成的协作规则整理进 CLAUDE.md / skill 草稿里。
```

---

## 默认协作口径

```text
以后在这个项目里默认按这个方式协作：
- 小改动直接做
- 多文件修改先分析再动手
- 复杂需求先给计划
- 高风险任务先分析并保守推进
- 改完要说明改动、验证结果和残余风险
```

---

## 两个提醒

- 不要一开始就把所有插件和技能都用上
- 不要让 Claude 一口气生成完整项目，优先小步推进
