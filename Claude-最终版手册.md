# Claude 最终版手册

## 0. 超短版速查

### 一句话原则

先理解项目，再固定规则，再沉淀高频流程，最后做自动化。

### 最小配置

- 一个 `CLAUDE.md`
- 两个 skill：`implementation`、`review`
- 两个 agent：`implementer`、`reviewer`
- 一个简单 hook：提醒验证或总结

### 五层分工

- `CLAUDE.md`：长期规则
- `skill`：高频任务模板
- `agent`：复杂任务分工
- `plugin`：整套工作流
- `hook`：自动校验或收尾

### 新项目顺序

1. 让 Claude 理解项目
2. 生成最小版 `CLAUDE.md`
3. 设计 2-3 个高频 skill
4. 设计 1-2 个 agent
5. 固定默认协作方式

### 老项目顺序

1. 先摸清技术栈、目录、核心模块、命令、风险点
2. 再生成 `CLAUDE.md`
3. 再补 skills / agents / hooks
4. 流程稳定后再做 plugin / hook 自动化

### 空白项目顺序

1. 整理项目目标
2. 拆 MVP
3. 选技术方案
4. 设计目录结构
5. 生成 `CLAUDE.md`
6. 拆第一批任务
7. 从最小骨架开始实现

### 最常用模板

接手项目：

```text
先不要改代码，帮我快速理解这个项目，并生成一份适合 Claude 协作的项目摘要。
```

做新功能：

```text
先分析需求会影响哪些模块，再给计划，等我确认后再改。
```

修 bug：

```text
先定位根因，不要急着修。给我最可能的原因、证据和最小修复方案。
```

做 review：

```text
请从 reviewer 角度检查这次改动，重点看回归风险、测试缺口和可维护性。
```

## 1. 核心原则

关键不是“装了多少插件和技能”，而是给 Claude 一个稳定的项目协作框架。否则能力很多，实际还是会乱用、漏用、重复用。

推荐主线：

1. 先让 Claude 理解项目
2. 再固定规则
3. 再沉淀高频流程
4. 最后才做自动化

### 推荐分层

- `CLAUDE.md`
  作用：项目长期规则

- `skills`
  作用：高频任务模板

- `agents`
  作用：复杂任务分工

- `plugins`
  作用：整套工作流

- `hooks`
  作用：自动校验或自动收尾

一句话：

- 规则写进 `CLAUDE.md`
- 经验沉淀成 `skill`
- 分工交给 `agent`
- 成熟流程打包成 `plugin`
- 重复动作交给 `hook`

## 2. 新项目或项目初期怎么用

目标不是一上来就自动化拉满，而是先把基础协作面搭起来。

### 先做这 4 件事

#### 1. 写一个最小版 `CLAUDE.md`

建议内容：

- 项目是干什么的
- 技术栈
- 目录结构
- 常用命令
- 代码规范
- 哪些文件不能乱动
- 希望 Claude 默认怎么工作

对话模板：

```text
请根据当前仓库生成一个最小可用的 CLAUDE.md。
要求包含：项目目标、技术栈、目录约定、开发命令、测试命令、改代码时的注意事项。
先分析仓库，再生成。
```

#### 2. 先定义 2-3 个最常用 skill / 工作流

优先高频场景：

- 代码审查
- 新功能实现
- bug 排查
- 文档生成

对话模板：

```text
基于这个项目，帮我设计 3 个最值得先有的 skills。
要求优先覆盖：功能开发、bug 修复、代码 review。
每个都说明用途、触发场景和建议结构。
```

#### 3. 给项目配 1-2 个 agent 就够

通常最先有价值的是：

- `implementation-agent`
- `reviewer`
- `debugger`

对话模板：

```text
请为这个项目设计两个最实用的 subagents。
一个负责实现功能，一个负责代码审查。
告诉我它们各自的职责边界和建议放进 .claude/agents/ 的定义方式。
```

#### 4. 固定默认协作方式

推荐默认节奏：

- 小任务：直接做
- 中任务：先读代码再改
- 大任务：先计划再改
- 高风险任务：先分析 + checkpoint
- 改完：说明改动 + 风险 + 验证结果

对话模板：

```text
以后在这个项目里，默认按这个方式协作：
小改动直接做；
多文件修改先分析相关文件再动手；
复杂需求先给计划；
改完要说明改了什么、有没有风险、做了哪些验证。
```

## 3. 老项目改造怎么用

老项目最容易出的问题是：Claude 没搞懂上下文就开始改。

推荐顺序：

1. 先让 Claude 建立项目认知
2. 再抽取规则到 `CLAUDE.md`
3. 再定义高频工作流
4. 最后才做更强自动化

### 推荐对话顺序

#### 第一轮：摸清项目

```text
先不要改代码。
请帮我快速摸清这个老项目：
1. 技术栈
2. 主要目录
3. 核心模块
4. 启动/测试方式
5. 明显的历史包袱或风险点
最后给我一个接手这个项目的摘要。
```

#### 第二轮：生成 `CLAUDE.md` 初稿

```text
根据你刚才的分析，帮我生成适合这个项目的 CLAUDE.md 初稿。
重点写：项目结构、常用命令、危险目录、改动原则、测试要求。
```

#### 第三轮：建立高频工作流

```text
基于这个老项目，建议我最先建立哪些 skills / agents / hooks。
要求按“投入低、收益高”排序。
```

## 4. 空白项目从 0 到 1

空白项目最好的用法不是让 Claude 直接从 0 一把梭，而是先搭出：

- 项目骨架
- 协作规则
- 初始节奏

Claude 最适合做 4 类事：

- 帮你澄清需求
- 帮你搭项目结构
- 帮你建立规则
- 帮你推进首批实现

### 推荐启动顺序

#### 第 1 步：先定义项目目标

```text
我要从 0 开始做一个新项目。
目标是：做一个给内部团队使用的任务管理 Web 应用。
用户是：产品和研发团队。
核心功能包括：登录、任务创建、任务分配、状态流转、评论。
请先帮我把需求整理成一个简洁清晰的项目说明，不要写代码。
```

#### 第 2 步：拆 MVP

```text
基于刚才这个项目说明，请帮我拆分 MVP。
要求分成：
1. 第一版必须有的功能
2. 可以第二阶段再做的功能
3. 暂时不做的功能
并说明这样拆分的理由。
```

#### 第 3 步：选技术方案

```text
现在请基于这个 MVP，帮我推荐技术方案。
请从开发速度、维护成本、部署难度、团队学习成本四个维度比较。
最后给出一个适合首版落地的建议栈。
```

#### 第 4 步：设计目录结构

```text
基于这个项目目标和技术选型，请设计一个适合 MVP 的项目目录结构。
要求：
1. 不要过度设计
2. 结构清晰
3. 便于后续扩展
4. 说明每个主要目录的职责
先出结构方案，不写代码。
```

#### 第 5 步：生成初始版 `CLAUDE.md`

```text
请根据我们刚才确定的项目目标、MVP 和技术方案，生成一个新项目初始版 CLAUDE.md。
要求包含：
1. 项目目标
2. 技术栈
3. 目录约定
4. 开发原则
5. 默认协作方式
6. 改代码时的约束
```

#### 第 6 步：拆第一批开发任务

```text
请基于当前方案，把项目的第一批开发任务拆成 8-12 个小任务。
要求：
1. 每个任务尽量独立
2. 有合理顺序
3. 标注前置依赖
4. 适合逐步实现和验证
```

#### 第 7 步：从骨架开始实现

```text
现在开始实现第一阶段。
先帮我完成项目初始化和基础目录搭建。
要求：
1. 保持最小可运行
2. 不要提前做没必要的抽象
3. 改完告诉我创建了哪些文件以及下一步建议
```

### 空白项目常见坑

- 一开始让 Claude 生成太多
- 没先明确 MVP
- 没建立规则
- 过早上 hooks / plugin

推荐开场话术：

```text
我要从 0 开始做一个新项目。
先不要写代码，先帮我完成这几件事：
1. 把我的想法整理成清晰的项目目标
2. 拆出 MVP 范围
3. 推荐合适的技术方案
4. 设计最小可扩展的目录结构
5. 生成项目初始版 CLAUDE.md
6. 把第一阶段开发任务拆出来

整个过程请遵循：
- 优先可落地，不要过度设计
- 优先 MVP，不提前做复杂扩展
- 每一步先解释思路，再继续下一步
```

## 5. 高频对话模板

### 接手一个已有项目

```text
先不要改代码。
请先帮我快速接手这个项目，输出：
1. 技术栈
2. 目录结构
3. 核心模块
4. 启动和测试方式
5. 风险点和历史包袱

最后给我一份“我接下来应该先做什么”的建议列表。
```

### 做一个新功能

```text
我们现在要做一个新功能。
先不要直接改代码。

请先完成这几步：
1. 找出相关模块和文件
2. 分析实现思路和风险点
3. 给一个尽量小步推进的计划

等我确认后再开始实现。
实现时要求：
- 尽量少改文件
- 保持现有代码风格
- 不做不必要的抽象

改完后请说明：
1. 改了哪些文件
2. 为什么这样改
3. 做了哪些验证
4. 还有哪些残余风险
```

### 修一个 bug

```text
先不要改代码。
帮我定位这个 bug：
[这里写现象]

请按下面格式输出：
1. 最可能的根因
2. 相关文件
3. 建议先验证什么
4. 最小修复方案

等我确认后再开始改。
改动时要求优先降低回归风险，尽量少改。
```

### 做一次代码 review

```text
请从 reviewer 的角度检查这次改动。
重点看：
1. 行为回归风险
2. 边界情况
3. 测试缺口
4. 可维护性问题

先列问题，再给建议。
如果没有明显问题，也请告诉我还存在哪些残余风险。
```

### 沉淀规则到 `CLAUDE.md`

```text
把这次我们形成的协作规则整理进 CLAUDE.md 草稿。
重点补充：
1. 项目约定
2. 默认协作方式
3. 常用命令
4. 改代码时的边界和注意事项
5. 验证要求
```

### 默认协作口径

```text
以后在这个项目里默认按这个方式协作：
- 小改动直接做
- 多文件修改先分析再动手
- 复杂需求先给计划
- 高风险任务先分析并保守推进
- 改完要说明改动、验证结果和残余风险
```

## 6. 项目模板合集

### 6.1 通用项目模板

适用场景：

- 暂时不想区分前后端
- 项目规模还不大
- 想先有一套稳妥通用的协作模板

#### 推荐的 `CLAUDE.md` 结构

```md
# 项目协作规范

## 项目目标
- 这个项目是做什么的
- 当前阶段目标是什么

## 技术栈
- 使用的主要框架、语言、工具

## 目录约定
- 主要目录分别负责什么
- 哪些目录不要轻易调整

## 常用命令
- 启动命令
- 测试命令
- 构建命令
- lint / format 命令

## 开发原则
- 优先小步修改
- 尽量少改文件
- 保持现有风格
- 不做不必要抽象

## 协作方式
- 小改动直接做
- 多文件修改先分析再动手
- 复杂任务先给计划
- 高风险任务先说明风险和方案

## 修改要求
- 改完说明修改点
- 说明验证结果
- 说明残余风险
```

#### 建议先建立的 skills

- `implementation`
- `review`
- `bugfix`

#### 建议先建立的 agents

- `implementer`
- `reviewer`

### 6.2 前端项目模板

适用场景：

- Web 前端
- React / Vue / Next.js / Nuxt 等项目
- 组件、交互、样式、页面结构较重要

#### 默认协作口径

```text
以后在这个前端项目里默认按这个方式协作：
- 改页面和组件前，先阅读现有结构和样式模式
- 优先复用现有组件和工具函数
- 不随意新增复杂抽象
- 样式和交互改动要说明影响范围
- 改完要说明界面行为、边界情况和验证结果
```

#### 做新功能模板

```text
我们现在要做一个前端功能。
先不要直接改代码。

请先输出：
1. 会影响哪些页面、组件、状态和接口
2. 推荐的实现路径
3. 是否需要新增组件或抽离公共逻辑
4. 视觉和交互风险点

等我确认后再实现。
实现要求：
- 优先复用现有组件
- 保持现有视觉风格和响应式行为
- 不做过度抽象

改完后请说明：
1. 改了哪些页面或组件
2. 用户可见变化是什么
3. 是否存在交互回归风险
4. 做了哪些验证
```

#### 建议先建立的 skills

- `frontend-implementation`
- `frontend-review`
- `frontend-bugfix`

#### 建议先建立的 agents

- `ui-implementer`
- `frontend-reviewer`
- `layout-debugger`

### 6.3 后端项目模板

适用场景：

- API 服务
- 管理后台服务端
- 微服务、数据处理、任务系统

#### 默认协作口径

```text
以后在这个后端项目里默认按这个方式协作：
- 改接口和数据逻辑前先分析影响范围
- 优先保持现有契约和兼容性
- 优先最小修改，避免无关重构
- 涉及数据库、缓存、消息时要说明一致性风险
- 改完要说明验证结果、兼容性影响和残余风险
```

#### 做新功能模板

```text
我们现在要做一个后端功能。
先不要直接改代码。

请先输出：
1. 相关模块、接口、数据表或任务
2. 推荐实现路径
3. 契约变化和兼容性影响
4. 数据一致性、错误处理和日志风险

等我确认后再实现。
实现要求：
- 优先最小改动
- 保持现有接口风格
- 不做无关重构

改完后请说明：
1. 改了哪些模块
2. 接口或数据行为有什么变化
3. 做了哪些验证
4. 还存在哪些兼容性或一致性风险
```

#### 建议先建立的 skills

- `backend-implementation`
- `backend-review`
- `backend-bugfix`

#### 建议先建立的 agents

- `api-implementer`
- `backend-reviewer`
- `incident-debugger`

## 7. 可直接复制的 `CLAUDE.md` 初稿

下面这份适合作为大多数项目的起始版 `CLAUDE.md`。建议先直接使用，再根据具体项目逐步补充。

```md
# 项目协作规范

## 项目目标

- 这个项目的核心目标是：
- 当前阶段最重要的任务是：
- 当前优先级高于长期优化，先保证可落地和稳定推进。

## 技术栈

- 主要语言：
- 主要框架：
- 包管理工具：
- 构建工具：
- 测试工具：

## 目录约定

- `src/` 或主业务目录：核心业务代码
- `tests/`：测试代码
- `docs/`：项目文档
- `scripts/`：辅助脚本
- 其他关键目录请根据项目实际情况补充说明

如果目录职责不明确，先阅读相关代码再决定修改位置，不要凭感觉新建结构。

## 常用命令

- 开发启动：
- 测试：
- 构建：
- lint：
- format：

如果不确定命令，先查项目配置，不要猜。

## 默认协作方式

- 小改动可以直接做
- 多文件修改先分析相关文件和影响范围
- 复杂需求先给计划，再开始改
- 高风险任务先说明风险和保守方案
- 修改完成后必须说明改动、验证结果和残余风险

## 开发原则

- 优先小步推进，不要一次改太多
- 优先最小可行修改，不做无关重构
- 尽量少改文件，保持现有结构和风格
- 优先复用现有模式，不随意引入新抽象
- 如果发现需求牵涉范围过大，要先提示，再决定是否继续

## 修改前要求

- 先理解相关模块和调用链
- 不清楚的地方先分析，不要直接写代码
- 涉及多个模块时，先说明影响范围
- 涉及高风险逻辑时，优先给最小改动方案

## 修改后要求

- 说明修改了哪些文件
- 说明为什么这样改
- 说明做了哪些验证
- 说明还存在哪些残余风险
- 如未运行测试或验证，必须明确说明

## 测试与验证

- 能跑相关测试时，优先跑最相关的测试
- 如果没有自动化测试，也要说明建议的手动验证方式
- 不要声称“已经修好”而不说明验证依据

## 风险控制

- 不要随意改动不相关模块
- 不要引入与当前任务无关的大规模重构
- 不要破坏现有接口、行为或目录约定，除非明确说明并得到确认
- 涉及删除、迁移、批量修改时，先说明风险

## 代码审查口径

在 review 或自检时，默认重点关注：

- 行为回归风险
- 边界情况
- 测试缺口
- 可维护性问题
- 不必要复杂度

## 适合直接使用的对话风格

在这个项目里，优先使用以下协作方式：

1. 新功能
   - 先分析模块和风险
   - 再给计划
   - 确认后再实现

2. 修 bug
   - 先定位根因
   - 再给最小修复方案
   - 确认后再修改

3. 做 review
   - 先列问题
   - 再给建议
   - 明确残余风险或测试缺口
```
