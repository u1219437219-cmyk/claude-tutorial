## 任务推进规则

- 小改动可以直接做
- 多文件修改先分析前后端影响范围
- 复杂需求先给计划，再开始改
- 跨前后端需求先拆成前端部分和后端部分
- 如果环境里已经安装 `superpowers`，优先使用它提供的流程型 skills；本文件和项目内 `.claude/` 规则继续作为项目约束
- 如果环境里已经安装 `claude-md-management`，在项目规则、目录、命令、技术栈或协作方式发生变化后，主动审计并更新本文件
- 如果环境里已经安装 `context7`，在查框架、库、工具链、配置项和版本相关 API 时，优先使用它获取最新官方文档和示例
- 如果环境里已经安装 `code-review`，在 PR 或合并前优先用它做自动化复查，但不要跳过项目内已有的前后端 review 口径
- 如果环境里已经安装 `security-guidance`，在编辑和实现阶段优先接受它的安全提醒，尽早修正明显的安全风险
- 如果环境里已经安装 `planning-with-files`，在复杂需求、长任务、跨前后端任务或多次会话推进时，优先把计划持久化到文件中并持续更新状态
- 如果环境里已经安装 `webapp-testing`，在前后端联调、关键页面流程验证和页面回归检查时，优先用它补真实页面验证
- 修改完成后必须说明改动、验证结果和残余风险

## 协作分层

- 模型本身负责理解、推理和行动，本文件和项目内 skills / agents / plugins 负责提供项目协作所需的 harness
- harness 的重点是给模型稳定的项目环境，包括规则、目录、技术栈、前后端边界、工具入口和验证方式
- 不要把临时流程堆叠误当成能力增强；优先保证规则清晰、边界稳定、验证可落地

- `Prompt / Command` 负责定义本次任务目标，比如实现、修 bug、review、先出计划再执行
- `Flow Skill` 负责决定整体方法，比如 `superpowers`、`planning-with-files`
- `Agent` 负责角色分工，比如 `frontend-implementer`、`backend-reviewer`
- `Project Skill` 负责具体执行口径，比如 `frontend-feature`、`backend-bugfix`、`frontend-review`
- `Plugin / Tool` 负责专项增强，比如文档查询、安全提醒、PR 复查
- 本文件负责项目级约束，比如目录、技术栈、边界、默认链路、输出要求

- 如果多层能力同时存在，优先顺序是：任务目标 -> 流程方法 -> 项目角色 -> 项目执行 skill -> 专项插件 -> 项目规则输出
- 不要让专项插件替代项目规则，也不要让项目规则替代流程方法

## 默认执行链路

- 默认执行顺序是：澄清需求 -> 拆边界 -> 落计划 -> 分边实现 -> 验证 -> 修正 -> 总结
- 如果是跨前后端任务，`拆边界` 和 `落计划` 不能省略
- 如果是高风险任务，`验证` 和 `修正` 不能省略
- 如果任务跨多轮会话，`落计划` 默认使用计划文件承接，而不是只靠对话上下文

### 各阶段含义

- `澄清需求`：确认目标、范围、限制条件和成功标准
- `拆边界`：区分前端部分、后端部分、接口契约、联动风险
- `落计划`：形成可执行步骤，并在需要时写入计划文件
- `分边实现`：前端按前端规则推进，后端按后端规则推进
- `验证`：跑最相关的测试、手动检查关键链路、确认契约和用户可见行为
- 涉及真实页面链路、联调流程或用户关键操作路径时，优先补一轮 `webapp-testing`
- `修正`：根据验证结果、安全提醒、review 发现的问题补修
- `总结`：输出改动、验证依据、残余风险，以及必要的 `CLAUDE.md` 更新建议

### 对应关系

- `澄清需求`：优先由自然语言指令和流程型 skill 驱动
- `拆边界`：优先由项目规则和前后端角色分工驱动
- `落计划`：优先由 `superpowers` 或 `planning-with-files` 承接
- `分边实现`：优先由 `frontend-*` / `backend-*` skills 和 agents 承接
- `验证`：优先由项目内 review 口径、测试命令和专项插件共同承接
- `验证` 阶段里，真实页面流程优先由 `webapp-testing` 承接
- `修正`：优先根据 `security-guidance`、项目内 review 和 `code-review` 结果补修
- `总结`：统一按本文件的输出要求回收

## 通用开发原则

- 优先小步推进
- 优先最小可行修改
- 不做无关重构
- 保持现有结构和风格
- 前后端联动改动要明确边界和依赖
- 流程型技能和项目型技能同时存在时，优先让流程型技能决定方法，再让项目型技能补充技术栈、目录和边界细节
- 需要引用外部技术资料时，优先查官方文档和权威示例，不凭模糊记忆猜测框架用法
- 安全问题尽量在实现阶段暴露和修正，不把明显的注入、XSS、权限和输入校验风险留到合并前再发现
- 多阶段任务尽量有可恢复的计划载体，避免跨会话后只靠聊天记录恢复上下文
- 跨前后端、预计超过一轮会话、需要分阶段推进或风险较高的任务，默认需要计划文件，不只在对话里口头列计划

## 计划文件要求

- 如果环境里有 `planning-with-files`，优先用它建立和维护计划文件
- 下列任务默认需要计划文件：跨前后端任务、长任务、多阶段任务、多会话任务、高风险任务
- 计划文件至少要包含：目标、边界、阶段拆分、当前状态、下一步、风险或阻塞点
- 每完成一个阶段后更新一次计划状态
- 重新开始一个中断过的任务时，先回读计划文件，再继续实现，不要只根据聊天记录恢复上下文

## 长任务默认产出物

- 新项目、长任务、跨前后端任务或高风险任务，除计划文件外，默认还应尽量形成可复用的产出物
- 不要求每次都写成完整长文档，但至少要让后续协作可以复用，而不是只留在聊天记录里

推荐优先产出：

- 规划文档：说明目标、范围、边界、阶段拆分和关键决策
- 接口或契约说明：说明前后端字段、错误处理、联动约束或数据流
- 前端交互 / 风格说明：在页面变化较大时，说明设计方向、交互原则或关键界面约束

简单任务不必强行补齐全部产出物。

但如果出现下面任一情况，至少补一类：

- 会跨多轮会话推进
- 会由不同人或不同 agent 接手
- 会同时影响前端和后端
- 会改变接口、交互或关键业务规则

## Hook 使用规则

- 当前项目默认使用轻量 hook，不要一开始就堆太多自动化
- hook 的目标是提醒、校验和兜底，不是替代人工判断
- 如果后续扩展 hook，优先按分级思路维护，而不是不断在一个文件里追加临时逻辑
- 推荐分级思路：
  - `minimal`：只保留关键提醒，比如修改后提示补充验证和风险
  - `standard`：增加常规检查，比如基础格式、测试提醒、计划状态提醒
  - `strict`：只在团队已经稳定适应后再启用，更强约束、更多校验、更严格阻断
- 默认从 `minimal` 起步，确认收益稳定后再升级到更高等级
- 如果 hook 开始明显打断开发节奏、误报过多或维护成本过高，应先降级或简化，而不是继续叠加
- 引入新 hook 时，要先说明它解决什么问题、会增加什么成本、什么时候应该触发

## 技术栈与外部资料使用

- 如果环境里有 `context7`，优先用它查询框架、库、工具链和版本相关 API 的官方资料
- 不确定的外部技术细节，先查官方文档和权威示例，再决定实现方式
- 不要凭模糊记忆猜测框架、库或配置项的当前行为

## 前端规则

- 改页面和组件前先阅读现有结构和样式模式
- 优先复用现有组件、hooks 和工具函数
- 涉及页面视觉风格、设计语言、落地页、仪表盘或整体 UI 美化时，优先使用 `frontend-design` skill
- 涉及普通页面功能、交互逻辑、状态处理和小范围样式调整时，优先使用 `frontend-feature` skill
- 说明用户可见变化、交互风险和响应式影响

## 后端规则

- 改接口和数据逻辑前先分析契约和兼容性影响
- 涉及数据库、缓存、消息、事务时先说明一致性风险
- 说明日志、错误处理和验证方式

## 安全规则

- 如果环境里有 `security-guidance`，实现过程中优先处理它给出的安全提醒
- 不把明显的注入、XSS、权限或输入校验问题留到 PR 阶段再集中处理
- 安全敏感改动要同时关注前端输入处理和后端校验/执行边界

## 修改后输出要求

- 说明前端改动了哪些页面、组件、状态或样式
- 说明后端改动了哪些接口、服务、数据逻辑或任务
- 说明联动影响
- 说明做了哪些验证
- 说明残余风险
- 如果本次协作暴露出 `CLAUDE.md` 缺失关键信息，补充更新建议，必要时使用 `claude-md-improver` 或 `/revise-claude-md`

## 审查与验证规则

- 日常 review、自检和定向检查，优先使用项目内的 `frontend-review` / `backend-review`
- PR 或合并前的自动化复查，如果环境支持，优先补一轮 `code-review`
- 默认顺序是：实现时先处理 `security-guidance` 的安全提醒，完成后按项目内 review 口径自检，PR 或合并前再补 `code-review`
- 涉及输入处理、权限、接口契约、数据库写入、富文本或 HTML 渲染的改动，必须同时经过安全检查和项目内 review 口径
- 如果 `security-guidance` 和项目内 review 都发现问题，优先先修安全风险，再处理结构、可维护性或测试缺口问题

### 前端重点

- UI 回归
- 交互一致性
- 组件边界
- 状态复杂度
- 样式和响应式影响

### 后端重点

- 接口契约
- 数据一致性
- 错误处理和日志
- 并发 / 事务 / 缓存 / 消息风险
- 测试缺口
