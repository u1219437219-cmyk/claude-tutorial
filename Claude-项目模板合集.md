# Claude 项目模板合集

## 用法说明

这份文件给出 3 套模板：

- 通用项目模板
- 前端项目模板
- 后端项目模板

每套都包含：

- 推荐的 `CLAUDE.md` 结构
- 推荐的默认协作口径
- 做新功能 / 修 bug / 做 review 的对话模板
- 建议先建立的 skills / agents

---

## 一、通用项目模板

适用场景：

- 暂时不想区分前后端
- 项目规模还不大
- 想先有一套稳妥通用的协作模板

### 推荐的 `CLAUDE.md` 结构

```md
# 项目协作规范

## 项目目标
- 这个项目是做什么的
- 当前阶段目标是什么

## 技术栈
- 使用的主要框架、语言、工具

## 目录约定
- 主要目录分别负责什么
- 哪些目录不要轻易调整

## 常用命令
- 启动命令
- 测试命令
- 构建命令
- lint / format 命令

## 开发原则
- 优先小步修改
- 尽量少改文件
- 保持现有风格
- 不做不必要抽象

## 协作方式
- 小改动直接做
- 多文件修改先分析再动手
- 复杂任务先给计划
- 高风险任务先说明风险和方案

## 修改要求
- 改完说明修改点
- 说明验证结果
- 说明残余风险
```

### 默认协作口径

```text
以后在这个项目里默认按这个方式协作：
- 小改动直接做
- 多文件修改先分析相关文件再动手
- 复杂需求先给计划
- 高风险任务先分析并保守推进
- 改完要说明改动、验证结果和残余风险
```

### 做新功能的模板

```text
我们现在要做一个新功能。
先不要直接改代码。

请先完成这几步：
1. 找出相关模块和文件
2. 分析实现思路和风险点
3. 给一个尽量小步推进的计划

等我确认后再开始实现。
实现时要求：
- 尽量少改文件
- 保持现有代码风格
- 不做不必要的抽象

改完后请说明：
1. 改了哪些文件
2. 为什么这样改
3. 做了哪些验证
4. 还有哪些残余风险
```

### 修 bug 的模板

```text
先不要改代码。
帮我定位这个 bug：
[这里写现象]

请按下面格式输出：
1. 最可能的根因
2. 相关文件
3. 建议先验证什么
4. 最小修复方案

等我确认后再开始改。
改动时要求优先降低回归风险，尽量少改。
```

### 做 review 的模板

```text
请从 reviewer 的角度检查这次改动。
重点看：
1. 行为回归风险
2. 边界情况
3. 测试缺口
4. 可维护性问题

先列问题，再给建议。
如果没有明显问题，也请告诉我还存在哪些残余风险。
```

### 建议先建立的 skills

- `implementation`
  作用：标准功能实现流程

- `review`
  作用：代码审查和风险检查

- `bugfix`
  作用：先定位根因，再最小修复

### 建议先建立的 agents

- `implementer`
  负责：功能实现和小步推进

- `reviewer`
  负责：审查回归风险、测试缺口、可维护性

## 二、前端项目模板

适用场景：

- Web 前端
- React / Vue / Next.js / Nuxt 等项目
- 组件、交互、样式、页面结构较重要

### 推荐的 `CLAUDE.md` 结构

```md
# 前端项目协作规范

## 项目目标
- 产品目标
- 当前版本目标

## 技术栈
- 框架
- 状态管理
- 样式方案
- 构建工具
- 测试工具

## 页面和组件约定
- 页面目录结构
- 组件目录结构
- 公共组件与业务组件的边界
- 样式组织方式

## 数据与状态约定
- 接口请求放哪里
- 全局状态和局部状态怎么分
- 副作用处理方式

## UI 和交互原则
- 优先复用现有组件
- 避免无意义复杂交互
- 保持视觉和交互一致性
- 不破坏现有布局和响应式行为

## 常用命令
- dev
- build
- lint
- test

## 协作方式
- 改 UI 前先读现有页面和组件模式
- 新组件先说明放在哪一层
- 样式改动要说明影响范围
- 改完要说明交互和视觉风险
```

### 默认协作口径

```text
以后在这个前端项目里默认按这个方式协作：
- 改页面和组件前，先阅读现有结构和样式模式
- 优先复用现有组件和工具函数
- 不随意新增复杂抽象
- 样式和交互改动要说明影响范围
- 改完要说明界面行为、边界情况和验证结果
```

### 做新功能的模板

```text
我们现在要做一个前端功能。
先不要直接改代码。

请先输出：
1. 会影响哪些页面、组件、状态和接口
2. 推荐的实现路径
3. 是否需要新增组件或抽离公共逻辑
4. 视觉和交互风险点

等我确认后再实现。
实现要求：
- 优先复用现有组件
- 保持现有视觉风格和响应式行为
- 不做过度抽象

改完后请说明：
1. 改了哪些页面或组件
2. 用户可见变化是什么
3. 是否存在交互回归风险
4. 做了哪些验证
```

### 修 bug 的模板

```text
先不要改代码。
帮我分析这个前端问题：
[这里写现象]

请按下面格式输出：
1. 最可能原因
2. 相关页面 / 组件 / 状态逻辑
3. 建议先验证的地方
4. 最小修复方案

等我确认后再开始修改。
优先修根因，避免用临时补丁掩盖问题。
```

### 做 review 的模板

```text
请从前端 reviewer 的角度检查这次改动。
重点看：
1. UI/交互回归风险
2. 组件边界是否清晰
3. 状态管理是否复杂化
4. 样式是否破坏现有布局或响应式
5. 测试或手动验证是否充分
```

### 建议先建立的 skills

- `frontend-implementation`
  作用：页面、组件、状态、接口联动实现

- `frontend-review`
  作用：检查 UI、交互、状态和样式风险

- `frontend-bugfix`
  作用：定位页面行为、事件、状态同步问题

### 建议先建立的 agents

- `ui-implementer`
  负责：页面和组件实现

- `frontend-reviewer`
  负责：UI/交互/状态风险审查

- `layout-debugger`
  负责：样式、布局、响应式问题定位

## 三、后端项目模板

适用场景：

- API 服务
- 管理后台服务端
- 微服务、数据处理、任务系统

### 推荐的 `CLAUDE.md` 结构

```md
# 后端项目协作规范

## 项目目标
- 服务解决什么问题
- 当前迭代目标是什么

## 技术栈
- 语言和框架
- 数据库
- 缓存
- 消息队列
- 测试框架

## 模块和目录约定
- 路由 / 控制器
- 服务层
- 数据访问层
- 定时任务 / 消费者
- 配置和基础设施目录

## 接口和数据原则
- 输入校验要求
- 错误处理方式
- 日志规范
- 返回结构约定
- 事务和数据一致性原则

## 常用命令
- 启动
- 测试
- lint
- migration

## 协作方式
- 改接口前先说明影响范围
- 涉及数据改动先说明风险
- 优先最小修复
- 改完要说明验证方式和兼容性影响
```

### 默认协作口径

```text
以后在这个后端项目里默认按这个方式协作：
- 改接口和数据逻辑前先分析影响范围
- 优先保持现有契约和兼容性
- 优先最小修改，避免无关重构
- 涉及数据库、缓存、消息时要说明一致性风险
- 改完要说明验证结果、兼容性影响和残余风险
```

### 做新功能的模板

```text
我们现在要做一个后端功能。
先不要直接改代码。

请先输出：
1. 相关模块、接口、数据表或任务
2. 推荐实现路径
3. 契约变化和兼容性影响
4. 数据一致性、错误处理和日志风险

等我确认后再实现。
实现要求：
- 优先最小改动
- 保持现有接口风格
- 不做无关重构

改完后请说明：
1. 改了哪些模块
2. 接口或数据行为有什么变化
3. 做了哪些验证
4. 还存在哪些兼容性或一致性风险
```

### 修 bug 的模板

```text
先不要改代码。
帮我分析这个后端问题：
[这里写现象]

请按下面格式输出：
1. 最可能根因
2. 涉及的接口 / 服务 / 数据层
3. 建议先验证的数据或日志
4. 最小修复方案

等我确认后再开始改。
优先修根因，并说明是否有数据一致性风险。
```

### 做 review 的模板

```text
请从后端 reviewer 的角度检查这次改动。
重点看：
1. 接口契约是否被破坏
2. 数据一致性风险
3. 错误处理和日志是否充分
4. 并发、缓存、事务或消息处理是否有隐患
5. 测试覆盖是否足够
```

### 建议先建立的 skills

- `backend-implementation`
  作用：接口、服务、数据层的标准实现流程

- `backend-review`
  作用：检查契约、一致性、错误处理和测试

- `backend-bugfix`
  作用：从日志、数据、调用链定位问题

### 建议先建立的 agents

- `api-implementer`
  负责：接口和业务实现

- `backend-reviewer`
  负责：契约、数据一致性、测试审查

- `incident-debugger`
  负责：日志、调用链、异常问题定位

## 四、如何选择用哪套模板

- 如果项目还在早期，或者全栈混合，先用“通用项目模板”
- 如果主要工作集中在页面、组件、样式和交互，用“前端项目模板”
- 如果主要工作集中在接口、服务、数据库和任务系统，用“后端项目模板”

最稳妥的做法通常是：

1. 先用“通用项目模板”起步
2. 项目稳定后，根据实际工作重心切换到“前端”或“后端”模板
