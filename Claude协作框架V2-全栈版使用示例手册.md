## 一、先记住一条总原则

在全栈项目里，最稳的方式不是一句话让 Claude 同时乱改前后端，而是：

1. 先分析这是前端问题、后端问题，还是跨前后端问题
2. 前端任务尽量调用 `frontend-*`
3. 后端任务尽量调用 `backend-*`
4. 跨前后端任务先拆边界，再分别推进

一句话：

**先分边，再处理。**

---

## 二、什么时候用哪类 skill / agent

### 前端类

适合这些任务：

- 页面改动
- 组件改动
- 样式问题
- 交互问题
- 状态同步问题

对应：

- `frontend-feature`
- `frontend-bugfix`
- `frontend-review`
- `frontend-implementer`
- `frontend-debugger`
- `frontend-reviewer`

### 后端类

适合这些任务：

- 接口修改
- 服务逻辑修改
- 数据写入问题
- 日志排查
- 任务系统问题

对应：

- `backend-feature`
- `backend-bugfix`
- `backend-review`
- `backend-implementer`
- `backend-debugger`
- `backend-reviewer`

### 跨前后端类

适合这些任务：

- 新功能同时涉及页面和接口
- 页面展示错了，同时接口数据也可能有问题
- 一个需求既改前端交互又改后端契约

做法：

- 先按通用规则分析
- 再拆成前端部分和后端部分
- 最后再整体验证联动

## 三、最常用的实际对话模板

## 1. 做一个前端功能

```text
这次按 frontend-feature 的思路推进。

我们要做一个前端功能：
[这里写需求]

先不要直接改代码。
请先输出：
1. 会影响哪些页面、组件、状态和接口
2. 推荐的实现路径
3. 是否需要新增组件或抽离公共逻辑
4. UI 和交互风险点

等我确认后再实现。
实现时请按 frontend-implementer 的方式：
- 优先复用现有组件和 hooks
- 保持现有视觉和交互风格
- 尽量少改文件

改完后请说明：
1. 改了哪些文件
2. 用户可见变化是什么
3. 做了哪些验证
4. 是否还有 UI / 交互 / 响应式风险
```

---

## 2. 做一个后端功能

```text
这次按 backend-feature 的思路推进。

我们要做一个后端功能：
[这里写需求]

先不要直接改代码。
请先输出：
1. 会影响哪些接口、服务、数据表、任务或调用链
2. 推荐的实现路径
3. 是否有契约变化或兼容性影响
4. 数据一致性、日志、错误处理风险

等我确认后再实现。
实现时请按 backend-implementer 的方式：
- 优先最小改动
- 保持现有接口风格
- 不做无关重构

改完后请说明：
1. 改了哪些模块
2. 行为有什么变化
3. 做了哪些验证
4. 是否还有契约或一致性风险
```

---

## 3. 修一个前端 bug

```text
这次按 frontend-bugfix 和 frontend-debugger 的方式处理。

问题现象：
[这里写 bug]

先不要改代码。
请先输出：
1. 最可能的根因
2. 相关页面、组件、hooks、样式或状态逻辑
3. 建议先验证什么
4. 最小修复方案

等我确认后再开始修改。
优先修根因，不要只做表面补丁。
改完后请说明：
1. 改了哪些文件
2. 修复了什么
3. 怎么验证
4. 还有哪些 UI 或交互风险
```

## 5. 做前端 review

```text
这次按 frontend-review 和 frontend-reviewer 的标准检查。

请 review 这次前端改动，重点看：
1. UI 回归风险
2. 交互一致性
3. 状态管理是否复杂化
4. 样式和响应式是否受影响
5. 测试和手动验证是否充分

先列 findings，再给总结。
```

---

## 6. 做后端 review

```text
这次按 backend-review 和 backend-reviewer 的标准检查。

请 review 这次后端改动，重点看：
1. 接口契约是否被破坏
2. 数据一致性风险
3. 日志和错误处理是否充分
4. 并发 / 事务 / 缓存 / 消息是否有隐患
5. 测试覆盖是否足够

先列 findings，再给总结。
```

---

## 7. 做跨前后端功能

```text
这是一个跨前后端需求。
先不要直接改代码。

请先按全栈项目的方式拆分：
1. 前端需要改什么
2. 后端需要改什么
3. 接口契约会怎么变化
4. 联调和回归风险是什么

然后给一个分阶段计划：
- 第一阶段：后端还是前端先做
- 第二阶段：另一侧怎么跟进
- 第三阶段：怎么联调验证

实现时：
- 前端部分按 frontend-feature / frontend-implementer 的方式推进
- 后端部分按 backend-feature / backend-implementer 的方式推进

最后请单独说明：
1. 前端改动
2. 后端改动
3. 联动验证方式
4. 残余风险
```

## 8. 做跨前后端 bug 排查

```text
这是一个跨前后端问题，先不要急着改代码。

现象：
[这里写问题]

请先帮我判断：
1. 更可能是前端问题、后端问题，还是联动问题
2. 前端最值得先看的位置
3. 后端最值得先看的位置
4. 建议的排查顺序

然后分别用：
- frontend-debugger 的思路分析前端部分
- backend-debugger 的思路分析后端部分

最后给我一个最可能的整体根因判断。
```

## 四、推荐的全栈协作闭环

最推荐的节奏是：

1. 先分析任务属于前端、后端还是跨端
2. 用对应的 `feature` 或 `bugfix` skill 收敛任务
3. 用对应的 `implementer` 或 `debugger` agent 执行
4. 改完后用对应的 `reviewer` 或 `review` 再检查一次
5. 最后说明：
   - 前端影响
   - 后端影响
   - 联调验证
   - 残余风险

## 五、最容易踩的坑

### 1. 不分边，直接一句话让 Claude 改全栈

这样最容易导致：

- 前后端边界不清
- 改动范围失控
- 联调风险被忽略

### 2. 前端任务用后端口径 review

会忽略：

- UI 回归
- 交互问题
- 样式影响

### 3. 后端任务用前端口径 review

会忽略：

- 契约风险
- 数据一致性
- 并发和事务问题

### 4. 跨端改动不单独说明联动验证

这是全栈项目最常见的问题之一。

---

## 六、一句话总结

全栈项目里最重要的不是“同时启用很多能力”，而是：

**前端用前端那套，后端用后端那套，跨端任务先拆边界，再分别推进。**
